Surge stores user preferences in a JSON configuration file. You can edit this file manually or use the built-in Settings UI in the TUI (press `s` to open).

## Configuration File Location

The `settings.json` file is located in the application data directory:

<CodeGroup>

```bash Linux
~/.config/surge/settings.json
# Or $XDG_CONFIG_HOME/surge/settings.json if set
```

```bash macOS
~/Library/Application Support/surge/settings.json
```

```bash Windows
%APPDATA%\surge\settings.json
```

</CodeGroup>

<Tip>If the file doesn't exist, Surge will create it with default values on first run.</Tip>

## Directory Structure

Surge organizes its files according to OS conventions:

| Directory   | Purpose                           | Linux                        | macOS                                       | Windows                 |
| :---------- | :-------------------------------- | :--------------------------- | :------------------------------------------ | :---------------------- |
| **Config**  | `settings.json`                   | `~/.config/surge/`           | `~/Library/Application Support/surge/`      | `%APPDATA%\surge\`      |
| **State**   | Database (`surge.db`), auth token | `~/.local/state/surge/`      | `~/Library/Application Support/surge/`      | `%APPDATA%\surge\`      |
| **Logs**    | Timestamped `.log` files          | `~/.local/state/surge/logs/` | `~/Library/Application Support/surge/logs/` | `%APPDATA%\surge\logs\` |
| **Runtime** | PID file, port file, lock         | `$XDG_RUNTIME_DIR/surge/`¹   | `$TMPDIR/surge-runtime/`                    | `%TEMP%\surge\`         |

<Note>
  ¹ Falls back to `~/.local/state/surge/` when `$XDG_RUNTIME_DIR` is not set (e.g., Docker/headless environments).
</Note>

## Settings Structure

The configuration file is organized into three main categories:

```json
{
  "general": { ... },
  "network": { ... },
  "performance": { ... }
}
```

## General Settings

Application behavior and UI preferences.

<ResponseField name="general.default_download_dir" type="string" default="">
  Default directory where new downloads are saved.
  
  If empty, defaults to:
  1. `$XDG_DOWNLOAD_DIR` (if set and exists)
  2. `~/Downloads` (if exists)
  3. Current directory (fallback)
  
  ```json
  "default_download_dir": "/home/user/Downloads"
  ```
</ResponseField>

<ResponseField name="general.warn_on_duplicate" type="boolean" default="true">
  Show a warning when adding a download that already exists in the queue.
  
  Prevents accidentally downloading the same file multiple times.
  
  ```json
  "warn_on_duplicate": true
  ```
</ResponseField>

<ResponseField name="general.extension_prompt" type="boolean" default="false">
  Prompt for confirmation in the TUI when adding downloads via the browser extension.
  
  When `true`, downloads from the extension require manual approval in the TUI. When `false`, they're automatically queued (unless `warn_on_duplicate` triggers).
  
  ```json
  "extension_prompt": false
  ```
  
  <Note>In headless mode, this setting causes extension downloads to be rejected unless the extension sets `skip_approval`.</Note>
</ResponseField>

<ResponseField name="general.auto_resume" type="boolean" default="false">
  Automatically resume paused downloads when Surge starts.
  
  When enabled, all downloads with status `paused` will be resumed on startup.
  
  ```json
  "auto_resume": false
  ```
  
  <Tip>Override with `--no-resume` flag to temporarily disable auto-resume.</Tip>
</ResponseField>

<ResponseField name="general.skip_update_check" type="boolean" default="false">
  Disable automatic check for new versions on startup.
  
  ```json
  "skip_update_check": false
  ```
</ResponseField>

<ResponseField name="general.clipboard_monitor" type="boolean" default="true">
  Watch the system clipboard for URLs and prompt to download them.
  
  When enabled, Surge monitors the clipboard and shows a prompt when a URL is detected.
  
  ```json
  "clipboard_monitor": true
  ```
</ResponseField>

<ResponseField name="general.theme" type="integer" default="0">
  UI theme preference.
  
  **Values:**
  - `0` - Adaptive (follows system theme)
  - `1` - Light
  - `2` - Dark
  
  ```json
  "theme": 0
  ```
</ResponseField>

<ResponseField name="general.log_retention_count" type="integer" default="5">
  Number of recent log files to keep.
  
  Older log files are automatically deleted on startup. Set to `0` to keep all logs.
  
  ```json
  "log_retention_count": 5
  ```
</ResponseField>

## Network Settings

Connection and download behavior.

<ResponseField name="network.max_connections_per_host" type="integer" default="32">
  Maximum concurrent connections allowed to a single host.
  
  **Valid range:** 1-64
  
  Higher values increase download speed but may be throttled or blocked by some servers.
  
  ```json
  "max_connections_per_host": 32
  ```
</ResponseField>

<ResponseField name="network.max_concurrent_downloads" type="integer" default="3">
  Maximum number of downloads running simultaneously.
  
  **Valid range:** 1-10
  
  ```json
  "max_concurrent_downloads": 3
  ```
  
  <Warning>Changing this setting requires restarting Surge.</Warning>
</ResponseField>

<ResponseField name="network.user_agent" type="string" default="">
  Custom User-Agent string for HTTP requests.
  
  If empty, uses Surge's default User-Agent. Some servers require specific User-Agent strings.
  
  ```json
  "user_agent": "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36"
  ```
</ResponseField>

<ResponseField name="network.proxy_url" type="string" default="">
  HTTP/HTTPS proxy URL for all downloads.
  
  If empty, uses system proxy settings.
  
  ```json
  "proxy_url": "http://127.0.0.1:8080"
  ```
  
  **Supported formats:**
  - `http://proxy.example.com:8080`
  - `https://proxy.example.com:8080`
  - `http://user:pass@proxy.example.com:8080`
</ResponseField>

<ResponseField name="network.sequential_download" type="boolean" default="false">
  Download file pieces in strict order (Streaming Mode).
  
  Useful for previewing media files before download completes, but may be slower than parallel downloading.
  
  ```json
  "sequential_download": false
  ```
</ResponseField>

<ResponseField name="network.min_chunk_size" type="integer" default="2097152">
  Minimum size of a download chunk in bytes.
  
  **Default:** `2097152` (2 MB)
  
  Larger chunks reduce overhead but may waste bandwidth if connections stall.
  
  ```json
  "min_chunk_size": 2097152
  ```
  
  <Info>Value in bytes: `1048576` = 1 MB, `2097152` = 2 MB</Info>
</ResponseField>

<ResponseField name="network.worker_buffer_size" type="integer" default="524288">
  I/O buffer size per worker in bytes.
  
  **Default:** `524288` (512 KB)
  
  Affects memory usage and disk write performance.
  
  ```json
  "worker_buffer_size": 524288
  ```
</ResponseField>

## Performance Settings

Advanced tuning for reliability and speed optimization.

<ResponseField name="performance.max_task_retries" type="integer" default="3">
  Number of times to retry a failed chunk before giving up.
  
  Helps recover from temporary network errors or server timeouts.
  
  ```json
  "max_task_retries": 3
  ```
</ResponseField>

<ResponseField name="performance.slow_worker_threshold" type="float" default="0.3">
  Restart workers slower than this fraction of the mean speed.
  
  **Valid range:** 0.0-1.0
  
  Example: `0.3` means workers performing at less than 30% of average speed are restarted.
  
  ```json
  "slow_worker_threshold": 0.3
  ```
</ResponseField>

<ResponseField name="performance.slow_worker_grace_period" type="duration" default="5s">
  Time to wait before checking a worker's speed.
  
  Prevents premature restarts during connection establishment.
  
  ```json
  "slow_worker_grace_period": 5000000000
  ```
  
  <Info>Value in nanoseconds: `5000000000` = 5 seconds</Info>
</ResponseField>

<ResponseField name="performance.stall_timeout" type="duration" default="3s">
  Restart workers that haven't received data for this duration.
  
  Helps recover from stalled connections.
  
  ```json
  "stall_timeout": 3000000000
  ```
  
  <Info>Value in nanoseconds: `3000000000` = 3 seconds</Info>
</ResponseField>

<ResponseField name="performance.speed_ema_alpha" type="float" default="0.3">
  Exponential moving average smoothing factor for speed calculation.
  
  **Valid range:** 0.0-1.0
  
  Lower values = smoother but slower to react to speed changes.
  Higher values = more responsive but noisier.
  
  ```json
  "speed_ema_alpha": 0.3
  ```
</ResponseField>

```json settings.json

  "general": {
    "default_download_dir": "/home/user/Downloads",
    "warn_on_duplicate": true,
    "extension_prompt": false,
    "auto_resume": false,
    "skip_update_check": false,
    "clipboard_monitor": true,
    "theme": 0,
    "log_retention_count": 5

  "network": {
    "max_connections_per_host": 32,
    "max_concurrent_downloads": 3,
    "user_agent": "",
    "proxy_url": "",
    "sequential_download": false,
    "min_chunk_size": 2097152,
    "worker_buffer_size": 524288
  },
  "performance": {
    "max_task_retries": 3,
    "slow_worker_threshold": 0.3,
    "slow_worker_grace_period": 5000000000,
    "stall_timeout": 3000000000,
    "speed_ema_alpha": 0.3
  }
}
```

## Editing Settings

### Via TUI (Recommended)

1. Press `s` in the TUI to open Settings
2. Navigate with arrow keys
3. Press `Enter` to edit a value
4. Changes are saved immediately

### Via File Editor

1. Open `settings.json` in your preferred editor
2. Make changes following valid JSON syntax
3. Save the file
4. Restart Surge for changes to take effect

<Warning>
  Invalid JSON will prevent Surge from loading settings. Always validate your JSON before saving.
</Warning>

## Migration & Compatibility

On Linux, Surge automatically migrates configuration from the legacy flat layout (`~/.config/surge/*`) to the XDG-compliant structure on first run:

- `surge.db` → `~/.local/state/surge/surge.db`
- `token` → `~/.local/state/surge/token`
- `logs/` → `~/.local/state/surge/logs/`

Settings remain at `~/.config/surge/settings.json`.

<Note>
  Legacy "connections" and "chunks" sections in old configuration files are automatically migrated to the "network" section.
</Note>
